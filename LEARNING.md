# 代码库学习计划

## 第一步：生成代码库概览

运行以下命令，让 Claude 扫描项目并生成 `CLAUDE.md`：

```
/init
```

**目标：** 得到项目结构、核心模块、入口文件的整体地图。

**完成标志：** `CLAUDE.md` 生成完毕，能说出各顶层目录的职责。

---

## 第二步：梳理核心流程

带着问题提问，让 Claude 定位到具体文件和行号：

- 这个框架的请求生命周期是怎样的？从入口到处理完整走一遍
- session 模块是怎么管理连接的？
- cluster 和 registry 之间是什么关系？
- 一条消息从客户端进来，经过哪些模块，最终怎么处理？

**目标：** 理解主干数据流和模块边界。

---

## 第三步：逐模块深入

按以下顺序逐个模块学习（由核心到外围）：

1. `transport` — 网络传输层
2. `session` — 连接管理
3. `cluster` + `registry` — 集群与服务发现
4. `eventbus` — 事件总线
5. `cache` / `lock` — 基础设施
6. 其余模块

每个模块学完后，用自己的话总结，让 Claude 纠正：

```
帮我总结 transport 模块的设计，我来说说我的理解，你纠正
```

**完成标志：** 能画出模块间的依赖关系图。

---

## 第四步：动手练习

每理解一个模块后，做一个小改动练习，不超过 20 行：

```
给我一个针对 lock 模块的小练习，不超过 20 行改动
```

**目标：** 通过修改代码验证理解，而不只是阅读。

---

## 第五步：结合二次开发目标

学习完核心模块后，针对自己的改动需求提问：

- 我想在 xxx 模块新增 yyy 功能，应该从哪里入手？
- 这个设计有什么扩展点？

---

## 常用提问模式

| 场景 | 提问方式 |
|------|----------|
| 找实现位置 | "eventbus 的发布订阅在哪里实现的？" |
| 读懂代码片段 | 贴代码，问"这段在做什么，为什么这样设计" |
| 了解模块边界 | "哪些模块会调用 session？依赖关系是什么" |
| 追踪数据流 | "一条消息从客户端进来，经过哪些模块" |
| 验证理解 | "我理解 xxx 是这样工作的，对吗？" |
